## 2. 集成结论

| 选型项 | 结论 | 原因 |
| --- | --- | --- |
| 应用模式 | 多智能体模式 | 学习闭环需要诊断、讲解、测评、规划、资料治理明确分工 |
| 协同方式 | 工作流编排 | 学生主链需要稳定、固定、可控 |
| 架构立场 | `ADP + Python + PostgreSQL` 混合方案 | ADP 负责推理，Python 服务更贴近平台生态，外部服务继续负责业务真源 |
| 记忆策略 | 长期记忆 + 外部业务状态 | 记忆负责连续感，数据库负责真状态 |
| 知识治理 | Git 化知识演化模型 | 解决上传资料后的可信度判断、发布和回滚 |
| 平台协作模式 | 用户先登录 ADP 控制台，AI 再接手浏览器操作 | 控制台配置、调试和发布依赖浏览器登录态，不在仓库存账号口令 |

## 3. 为什么不用“全都塞进 ADP”

ADP 足够强，但下面几类状态不应该只存在于上下文或长期记忆里：

- 学习地图
- 节点状态
- 画像分值
- 复习计划
- 知识版本
- 发布与回滚记录

正式原则：

- `ADP` 负责多智能体、工作流、知识检索、流式输出、文档解析
- `Python + PostgreSQL` 负责学习状态、知识版本、冲突门禁、回滚、影响域重算

## 3.1 ADP 平台协作前置

只要任务涉及 ADP 控制台、工作流配置、发布检查或浏览器内调试，统一遵守下面这条协作约束：

- 由你先完成腾讯平台登录并保持会话有效
- 我再接手浏览器做配置、核查、联调和截图取证
- 如果会话失效、触发验证码或需要二次确认，任务立即停在“等待人工登录恢复”
- 仓库和本地脚本只保存路径、项目标识和非敏感配置，不保存平台账号口令

## 4. 正式工作流

### 4.1 学生主链

`选科 -> 初始地图 -> 短诊断 -> 当前关卡教学 -> 作答评分 -> 地图重规划 -> 画像更新 -> 笔记生成`

### 4.2 新资料链路

`上传资料 -> COS -> SaveDoc -> 解析/OCR -> 声明抽取 -> 冲突判断 -> 候选分支 -> 发布 -> 影响域重算`

### 4.3 为什么不用自由转交做主链

- 学生主链需要可控和可解释
- 学习地图必须能稳定落库
- 评分与重规划需要结构化回写

所以主链固定采用工作流编排；`自由转交` 只保留为探索性增强，不进入正式主链。

### 4.4 Plan & Execute 的位置

`Plan & Execute` 只用于离线或弱实时任务：

- 长目录生成
- 大批资料分析归类
- 阶段性策略重算
- 笔记汇编

不直接挂在学生实时闯关主链上。

## 5. 上下文与长期记忆策略

### 5.1 上下文只传局部状态

每次送入 ADP 的上下文默认只保留：

- 当前关卡卡片
- 当前地图局部视图
- 学习画像摘要
- 最近必要交互
- 当前检索出的知识片段

### 5.2 不送入模型的内容

- 全量学习地图
- 全量错题本
- 全量历史笔记
- 多学科完整目录

### 5.3 长期记忆分工

`VisitorId`：稳定用户标识  
`ConversationId`：一次学习会话标识  
`SYS.Memory`：长期摘要记忆  
`custom_variables`：课程、节点、风险、范围等运行时变量

长期记忆只存：

- 偏好
- 薄弱点
- 错误模式
- 节奏特征
- 兴趣点

长期记忆不存：

- 地图真状态
- 节点完成状态
- 知识提交状态
- 完整复习计划

## 6. Agent 与模型建议

| Agent | 角色 | 模型建议 |
| --- | --- | --- |
| `StarterAgent` | 生成初始地图 | 规划型模型 |
| `DiagnosisAgent` | 短诊断与起点判断 | 推理型模型 |
| `MapPlannerAgent` | 地图重规划 | 推理型模型 |
| `TutorAgent` | 流式讲解与引导 | 讲解型模型 |
| `EvaluatorAgent` | 评分与反馈 | 推理型模型 |
| `ProfileAgent` | 画像更新 | 稳定结构化输出模型 |
| `NoteSynthAgent` | 笔记与思维导图汇总 | 总结型模型 |
| `IngestionAgent` | 资料解析、声明抽取、补丁生成 | 解析与推理型模型 |
| `StrategyAgent` | 影响分析与策略快照 | 规划型模型 |

工程口径统一写成：

- 平台可按 Agent 分工选用 `DeepSeek` 与 `腾讯混元`
- 不把全链路固定到单模型
- 第三方模型能力以当前采购版本为准

## 7. Git 化知识治理

### 7.1 四个知识区

| 知识区 | 说明 |
| --- | --- |
| `raw` | 原始资料和 OCR 中间结果 |
| `candidate` | 候选知识区 |
| `main` | 主教学知识区 |
| `archive` | 归档与回滚区 |

### 7.2 核心对象

- `KnowledgePatch`
- `KnowledgeCandidate`
- `KnowledgeConflict`
- `KnowledgeCommit`
- `KnowledgeRelease`
- `KnowledgeRollback`
- `ValidationReport`
- `EvidenceBundle`

### 7.3 发布规则

- 原始资料先落 `raw`
- AI 先做声明抽取和 `KnowledgePatch`
- 校验通过后进入 `candidate`
- 满足门禁后生成 `KnowledgeCommit`
- 合并到 `main`
- 如有问题可 `rollback`

## 8. 资料入库为什么是异步链路

正式流程不是“上传即生效”，而是：

1. 前端上传资料
2. 后端申请临时凭证
3. 文件上传 COS
4. 调用 `SaveDoc`
5. 轮询解析状态
6. 生成知识补丁
7. 进入候选分支
8. 合并发布并重算影响域

这样才能同时保证可追溯、可回滚、可解释。

## 9. 集成约束

| 约束 | 说明 |
| --- | --- |
| 工作流优先稳定性 | 学生主链固定采用工作流编排 |
| 记忆依赖稳定标识 | 学生标识不稳定会直接破坏跨会话连续性 |
| 发布优先可信度 | 未校验资料不能进入主教学知识区 |
| 解析优先异步链路 | 资料上传后先走候选区，不承诺直接主干生效 |
| 高数先做示范 | 知识治理和门禁先用高数场景验证 |
