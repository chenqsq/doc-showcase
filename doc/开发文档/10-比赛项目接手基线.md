# 比赛项目接手基线

本文件给“第一次接手这个比赛仓的人”使用，目标不是讲概念，而是让你在最短时间内知道这个仓现在是什么、做到哪一步、接下来该从哪里开工。

## 1. 一眼看懂当前项目

| 项目 | 当前事实 |
| --- | --- |
| 仓库角色 | 比赛展示站 + 作品文档主线 + 工程参考层 + 调试知识库样例 |
| 当前主技术路线 | `ADP + Python + PostgreSQL` |
| 前端现状 | `React 18 + TypeScript + Vite` 展示站已可构建 |
| 智能体现状 | 已有 ADP Prompt 资产和知识库样例，但未在本仓形成完整运行态后端 |
| ADP 接口进度 | `2026-04-12` 已完成首次真实接口验证（最小 HTTP SSE V2：`HTTP 200` + 完整流式事件 + 完成信号） |
| 当前优先级 | 先保证比赛叙事、文档口径、演示链路一致，再推进 ADP 真接入 MVP |
| 浏览器协作前置 | 涉及 ADP 控制台时，必须由你先登录，我再接手浏览器操作 |

## 2. 当前仓库到底承担什么

这个仓不是“已经跑通全部后端与智能体编排的成品后端仓”，更像下面四层合在一起：

1. 比赛展示站  
   负责把作品文档、工程文档、知识库样例和平台资料组织成可展示、可答辩、可查阅的站点。
2. 作品主线文档  
   核心叙事、PRD、页面范围、技术选型、答辩口径都在 `doc/作品文档`。
3. 工程参考层  
   开发与接入规则、架构、调试、测试计划在 `doc/开发文档`。
4. 智能体资产样例  
   Prompt 资产在 `ops/adp-prompts/highmath-v1`，高数调试知识库样例在 `kb/高等数学_测试`。

## 3. 当前技术路线为什么改成 Python

这次接手基线把后端主路线从 `Go` 降成了 `Python`，原因不是 Go 不行，而是比赛阶段更需要“快出效果、贴平台生态、容易联调”。

| 维度 | 当前判断 |
| --- | --- |
| 腾讯 ADP 生态贴合度 | Python 更顺手，平台资料、代码节点和示例更贴近 Python 习惯 |
| 比赛交付速度 | Python 更适合快速补 Agent 编排胶水层、RAG、评测和平台调试脚本 |
| 业务真源承接 | 仍然保留 `PostgreSQL`，不把业务真状态只放在上下文或长期记忆 |
| 后续扩展 | 如果以后真的需要更强业务服务层，再补独立服务，不影响当前比赛版路线 |

## 6. 现在能直接做什么

### 6.1 当前已验证能力

- `npm run build` 可以通过
- 展示站能承载作品文档、开发文档、知识库与参考资料
- Prompt 资产和高数知识库样例已在仓内落位
- `2026-04-12` 已完成 ADP 首次真实接口验证：人工登录、测试应用发布可见、最小 HTTP SSE V2 请求 `HTTP 200` 且流式完成

### 6.2 当前还没真正落地的能力

- ADP 知识库真实接入与检索稳定性验证
- SaveDoc 通路验证
- 比赛级主场景端到端验收
- Python 后端运行时骨架
- PostgreSQL 真正落库的业务状态模型
- 后端/BFF 与 ADP 的稳定联调闭环

## 7. ADP 平台协作怎么配合

| 场景 | 协作方式 |
| --- | --- |
| 只读文档与站点改造 | 我可以直接处理 |
| ADP 控制台配置 | 你先登录平台并保持会话，我再接手浏览器操作 |
| 平台登录过期 | 任务暂停，等待你重新登录 |
| 账号口令与敏感信息 | 不写入仓库、不写入脚本、不写入文档正文 |

这条约束很重要，因为后面真正做 ADP 平台侧配置、工作流调试、发布检查时，浏览器登录态就是前置条件。

## 9. 下一步默认开工顺序

1. 基于 `2026-04-12` 首次真实接口验证结果，继续收紧作品主线与工程主线口径，明确“已验证”和“未验证”边界。
2. 在 ADP 侧继续补知识库接入、SaveDoc 通路与主场景链路验证（不暴露敏感值，旧密钥先旋转）。
3. 给 Python 后端补最小运行骨架，承接 ADP 外部业务真源并准备 BFF/后端联调。
4. 每次联调完成后，按证据回写测试文档和接手基线，不用口号替代证据。
