# AI主导学习生命周期的自进化自学智能体平台开发技术文档

> 文档层级：作品主文档  
> 文档目的：说明平台选择、关键实现、部署方式和二次开发边界  
> 核心结论：比赛版技术实现遵循“作品文档做唯一主线、ADP 负责智能体运行时、Python 单体负责业务真源、知识库按 Git 化模型演化”的原则

## 1. 技术实现结论

平台正式采用 `ADP + Python + PostgreSQL` 的混合方案：

- `Vue 3 + TypeScript` 承接学生页与后台页
- `Python 3.12 + FastAPI` 承接学习地图、画像、知识版本和业务 API
- `PostgreSQL 16` 承接学习状态、知识提交、审计与回滚
- `腾讯 ADP + HTTP SSE V2` 承接多智能体编排、知识检索、流式教学和文档解析

这条路线的重点不是“把所有事都塞进一个大模型会话”，而是把智能体推理和业务真状态拆开。

## 2. 为什么选腾讯 ADP

腾讯 ADP 当前最适合承接这类产品的几项能力是：

- 多智能体模式与工作流编排
- 流式输出与发布接入
- 文档解析、知识库检索与标签隔离
- 每个 Agent 按职责选择不同模型

但平台不把下面这些真状态只放在 ADP 里：

- `AI学习地图`
- `地图节点状态`
- `学习画像分值`
- `复习计划`
- `知识版本与回滚记录`

这些对象必须落在 Python 服务和 PostgreSQL 里，避免上下文漂移和长期记忆污染业务状态。

## 3. 混合架构分工

| 层 | 负责什么 |
| --- | --- |
| 学生前端 | 选科、学习地图、闯关学习、笔记成长展示 |
| 后台前端 | 资料注入、知识演化、策略分析和审计 |
| Python 单体服务 | 学习会话、地图状态、画像落库、知识治理、版本发布 |
| 腾讯 ADP | 多智能体运行时、工作流、知识检索、流式教学、资料解析 |
| PostgreSQL | 学习状态、知识补丁、知识提交、冲突记录、审计日志 |
| COS / MinIO | 原始资料、思维导图与导出资源 |

正式职责边界：

- `ADP` 负责推理、检索、生成和工作流协作。
- `Python + PostgreSQL` 负责状态真源、门禁、回滚和局部重算。

### 3.1 ADP 控制台协作前置

凡是涉及 ADP 控制台、工作流配置、浏览器调试、发布检查的任务，统一采用这个协作模式：

- 先由人工完成腾讯平台登录
- 再由 AI 接手浏览器做配置、核查和联调
- 会话失效或出现二次验证时，任务暂停等待人工恢复登录
- 平台账号口令不进入仓库、不进入脚本、不进入文档正文

## 4. 上下文与长期记忆边界

平台明确承认上下文有限，所以正式策略是“局部上下文 + 外部真状态 + 结构化回写”。

### 4.1 每次送入模型的上下文只保留

- 当前关卡卡片
- 当前地图局部视图
- 学习画像摘要
- 最近必要交互
- 当前检索命中的知识片段

### 4.2 不允许反复拼进模型的内容

- 全量学习地图
- 全量错题本
- 全量历史笔记
- 多门课全量目录

### 4.3 长期记忆只存摘要

`SYS.Memory` 只存：

- 偏好
- 薄弱点
- 常见错误模式
- 节奏特征
- 兴趣点

`SYS.Memory` 不存：

- 节点完成状态
- 地图真状态
- 完整复习计划
- 知识库版本信息

## 5. Agent 群与模型分工

| Agent | 最小职责 | 推荐模型取向 |
| --- | --- | --- |
| `StarterAgent` | 生成初始地图 | 规划型模型 |
| `DiagnosisAgent` | 执行短诊断 | 推理型模型 |
| `MapPlannerAgent` | 负责地图重规划 | 推理型模型 |
| `TutorAgent` | 负责讲解和引导 | 讲解型模型 |
| `EvaluatorAgent` | 负责评分和成长反馈 | 推理型模型 |
| `ProfileAgent` | 负责画像更新 | 结构化输出稳定型模型 |
| `NoteSynthAgent` | 负责笔记和思维导图 | 总结型模型 |
| `IngestionAgent` | 负责资料识别、声明抽取和补丁生成 | 解析与推理型模型 |
| `StrategyAgent` | 负责策略快照、影响分析和日志汇总 | 规划型模型 |

平台允许按 Agent 职责选用 `腾讯混元` 与 `DeepSeek`，而不是全链路只绑一个模型。

## 6. Git 化知识治理

平台不把“上传资料”直接等同于“主知识成立”，而是采用 Git 化知识演化模型。

### 6.1 四个知识区

| 知识区 | 作用 |
| --- | --- |
| `raw` | 原始资料与 OCR 中间结果 |
| `candidate` | 候选知识区，存放待验证知识补丁 |
| `main` | 主教学知识区，供正式学习与评分链路使用 |
| `archive` | 归档与回滚区 |

### 6.2 核心对象

- `KnowledgePatch`
- `KnowledgeCommit`
- `KnowledgeCandidate`
- `KnowledgeConflict`
- `KnowledgeRelease`
- `KnowledgeRollback`
- `ValidationReport`
- `EvidenceBundle`

### 6.3 正式发布规则

- 原始资料先进 `raw`
- AI 抽取知识声明并生成 `KnowledgePatch`
- 校验后进入 `candidate`
- 通过门禁后形成 `KnowledgeCommit`
- 合并到 `main`
- 出问题可 `rollback`

### 6.4 可信度判断

资料可信度判断至少覆盖：

- 来源可信度
- 解析质量
- 结构完整度
- 与主干知识一致性
- 交叉证据支持度
- 教学可用性

高风险或冲突资料不会直接进入主教学区。

## 7. 近实时更新策略

实时更新定义为两层：

- `candidate` 候选区实时可见
- `main` 主教学区近实时合并

知识变更后不做全量重建，而是只重算受影响范围：

- 受影响知识点
- 受影响章节
- 受影响地图节点
- 受影响学生的局部学习路径

正在学习中的学生默认不被当前关卡强行打断，而是在下一轮任务切换时接收更新。

## 8. 单机部署建议

### 单机结构

- 1 个 Vue 前端静态站点
- 1 个 Python 单体服务
- 1 个 PostgreSQL 数据库
- 1 个 COS / MinIO 文件存储

### 运行建议

- 使用 `Caddy / Nginx` 反向代理
- 使用 `systemd` 守护 Python 服务
- 为流式接口和后台任务保留健康检查
- 保留数据库备份、恢复和知识版本回滚脚本

## 9. 二次开发边界

### 允许扩展

- 新学科接入
- 多科排程策略增强
- 地图样式与动画升级
- 新的画像字段
- 新的笔记生成模板
- 更细的知识发布门禁

### 当前不前置

- 微服务拆分
- Redis / MQ
- 自动微调底层模型
- 把聊天上下文当数据库
- 未校验资料直写主教学知识区
