# 知脉课堂答辩 PPT 大纲

> 文档层级：交付文档  
> 文档目的：提供一套可直接制作 PPT 的 16 页结构  
> 核心结论：答辩 PPT 要先让评委看懂“问题 -> 方案 -> 页面 -> 技术 -> 效果 -> 交付”，不要一上来堆实现细节

## 1. 总体建议

- 建议页数：`16` 页左右
- 建议时长：`5` 到 `8` 分钟
- 优先用流程图、页面截图、结果示意，不堆大段文字

## 2. 页次结构

| 页码 | 标题 | 关键内容 |
| --- | --- | --- |
| 1 | 封面 | 作品名、团队名、学校、比赛名称 |
| 2 | 问题背景 | 课堂资料碎片化、复盘断裂、教师难看学情 |
| 3 | 解决思路 | “课堂重构 + 学生伴学 + 错题复练 + 教师洞察” 四段式方案 |
| 4 | 作品定位 | `知脉课堂` 的一句话定义与角色范围 |
| 5 | 五条场景链路 | 上传、重构、伴学、复练、洞察 |
| 6 | 页面总览 | 六个核心工作台页面 |
| 7 | 课堂资料上传与重构页 | 多模态素材进入系统后的结果展示 |
| 8 | 学生伴学会话页 | 流式讲解、作答与掌握度反馈 |
| 9 | 错题与复习页 | 错因标签、变式题、复习调度 |
| 10 | 教师洞察页 | 图表、风险学生、补讲建议 |
| 11 | 总体架构 | Vue 前端、Go 后端、ADP、PostgreSQL、COS |
| 12 | 关键算法 | 知识重构、任务规划、掌握度评分、错题变式、群体洞察 |
| 13 | 知识库与数据来源 | 教材、讲义、录音、板书、题库、错题 |
| 14 | 访问与 API | 访问方式、SSE、评委体验路径 |
| 15 | 预期效果与测试 | 标准链路、交付检查、预期价值 |
| 16 | 总结与展望 | 可扩学科、可接产品、后续迭代方向 |

## 3. 每页讲述提示

- 第 2 页只讲痛点，不讲技术栈
- 第 6 到 10 页是评委最容易形成记忆点的页面段
- 第 11 到 13 页用于证明“不是只做了个聊天界面”
- 第 14 页回答“怎么体验、怎么接入”
- 第 16 页收在“平台可扩展与教育价值”

## 4. 可直接复用的开场语

> 我们的作品叫知脉课堂，它面向高校课堂场景，把课堂录音、PPT、讲义、板书和学生拍题统一重构为可学习、可复练、可洞察的知识链路。它既服务学生，也服务教师，还保留了可接入真实产品的工程结构。

## 5. 可直接复用的收尾语

> 我们想证明的不只是 AI 能解一道题，而是它能否把课堂知识真正组织起来，持续帮助学生学习，并让教师及时看见问题、采取干预。
