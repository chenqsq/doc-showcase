# AI教师子引擎工作流联调与验收手册

> 文档层级：子引擎层  
> 文档目的：只保留 AI 工作流真正要联调的内容，包括角色边界、状态槽位、检索绑定、回归样例和通过标准  
> 核心结论：工作流稳定的关键不是提示词越多越好，而是角色边界清楚、状态统一、检索命中稳定、失败原因可复现  
> 目标读者：工作流与联调负责人、实现者  
> 推荐下一步：需要高数落地版本时看 [高等数学提示词与ADP实施.md](../学科层/高等数学提示词与ADP实施.md)

## 1. 6 个核心角色怎么分

| 角色 | 只负责什么 |
| --- | --- |
| `TeacherOrchestrator` | 接收请求，决定往哪条子流程走 |
| `DiagnosisAgent` | 判断学生当前状态、薄弱点和风险 |
| `ExplanationAgent` | 负责讲解和搭桥 |
| `PracticeEvalAgent` | 负责练习、纠错和达标判断 |
| `ReviewPlanAgent` | 负责复盘和下一步安排 |
| `TeacherOpsAgent` | 负责教师侧摘要和运营视角总结 |

任何角色一旦既做诊断又做总结又做复盘，工作流就会开始漂。

## 2. 状态槽位最少统一哪些

无论提示词怎么写，至少统一下面几类状态：

- 学生当前基础
- 当前目标模块
- 当前问题类型
- 当前是否需要补桥
- 当前是否进入练习或复盘
- 当前是否需要教师介入

这些槽位是平台对象、工作流和学科提示词之间的共同语言。

## 3. 路由规则只保留什么判断

最少只做这几类判断：

1. 先诊断还是直接解释
2. 当前是补桥、常规推进还是纠错
3. 是否需要调用教师侧摘要
4. 检索范围应该落在什么资产类型上

不要把路由写成一个难以维护的大条件树。

## 4. 变量透传怎么理解

变量透传不是为了“配置完整”，而是为了保证：

- 平台识别出来的状态能进入工作流
- 工作流产出的结果能回到平台
- 教师侧看到的是同一轮真实数据

每次联调至少确认这三条链路：

1. 平台 -> 子引擎
2. 子引擎 -> 知识资产检索
3. 子引擎 -> 平台 / 教师侧

## 5. 检索绑定最容易出什么问题

最常见的 4 种问题：

1. 标签不够细，导致命中太粗。
2. 关键词不自然，导致学生问法搜不到。
3. 资产类型选错，导致讲解和练习混用。
4. 学科层提示词和知识库命名不一致。

所以联调时一定要同时看：

- 资产标题
- 关键词与别名
- 资源类型
- 实际命中结果

## 6. 回归样例最少覆盖什么

1. `0基础补桥`
2. `常规提分`
3. `刷题纠错`
4. `教师运营摘要`
5. `检索不到时的降级回复`

每类样例都要保留：

- 输入问题
- 命中资产
- 返回结果
- 是否通过
- 如果不通过，原因是什么

## 7. 通过标准怎么写最有用

不要只写“效果好”。

建议只看这 5 项：

1. 路由是否正确。
2. 检索是否命中正确资产。
3. 回答是否符合当前状态。
4. 结构化结果是否回流成功。
5. 失败时是否能解释原因。

## 8. 读完后你应该带走什么

- 联调手册现在只负责工作流本身，不再重复平台背景。
- 如果工作流不稳，优先回查状态、路由和检索，不要只改措辞。
- 高数具体版本请继续看学科实施文档。
