# 比赛交付与答辩手册

> 文档层级：交付层  
> 文档目的：说明比赛版文档包怎么收口、怎么彩排、怎么给评委访问  
> 核心结论：交付层负责把作品文档、演示顺序、访问方式和责任分工收成一套稳定执行手册  
> 目标读者：项目负责人、材料负责人、主讲人、现场协作者  
> 推荐下一步：准备答辩先读 [答辩口径与演示脚本.md](./答辩口径与演示脚本.md)

## 与其他文档的边界

本文负责“怎么交”和“怎么讲”。  
平台结构、对象契约、子引擎实现和学科知识设计仍回到平台层、子引擎层和作品文档正文。

## 一句话先记住

> 交付层不是再写一遍技术方案，而是确保材料齐、口径稳、入口清、演示顺。

## 1. 当前交付包结构

比赛交付包统一收口到 `doc/作品文档`：

1. `00-作品总览与阅读地图`
2. `01-PRD与需求分析`
3. `02-场景与用户流程`
4. `03-页面与交互设计`
5. `04-总体架构与技术选型`
6. `05-算法与知识库设计`
7. `06-接口与API说明`
8. `07-测试验证与预期效果`
9. `08-答辩PPT大纲`
10. `09-演示视频脚本`
11. `10-访问与评测手册`
12. `11-开发技术文档`

## 2. 页数与时长控制

| 交付物 | 控制要求 |
| --- | --- |
| 方案说明书 | Word 20 页内 |
| 技术文档 | Word 10 页内 |
| PPT | 20 页内 |
| 视频 | 3 分钟内 |

## 3. 现场演示顺序

推荐固定顺序：

1. 课堂工作台首页
2. 课堂资料上传与重构页
3. 学生伴学会话页
4. 错题与复习页
5. 教师洞察页
6. 管理与接入配置页

说明：

- 顺序与作品文档、视频脚本、PPT 大纲保持一致
- 所有页面都要围绕“上传资料 -> 重构 -> 伴学 -> 复练 -> 洞察”主链展开

## 4. 团队分工建议

| 角色 | 主要职责 |
| --- | --- |
| 项目负责人 | 控制术语一致性、文档收口、答辩口径 |
| 前端负责人 | 页面实现、演示流畅性、状态可视化 |
| 后端负责人 | API、SSE、存储、权限与部署 |
| 算法负责人 | 知识重构、掌握度、错题画像、教师洞察 |
| 材料负责人 | PPT、视频脚本、访问手册与比赛格式收尾 |

## 5. 评委访问准备

评委体验前必须准备：

- 访问地址
- 账号密码或体验凭证
- 角色区分说明
- 操作步骤
- 异常说明
- API 摘要说明

这些内容统一回链到：

- [10-访问与评测手册.md](../../作品文档/10-访问与评测手册.md)
- [06-接口与API说明.md](../../作品文档/06-接口与API说明.md)

## 6. 彩排检查表

- 作品名是否统一为“知脉课堂”
- 6 个核心页面是否都能打开
- 演示链路是否能在 3 分钟视频版和 5 到 10 分钟答辩版之间切换
- 页面、PPT、视频脚本、访问手册中的术语是否一致
- 是否仍残留“开发文档入口”等旧表述

## 7. 交付层的固定职责

- 收口材料结构
- 收口页面演示顺序
- 收口访问说明
- 收口答辩话术

交付层不负责：

- 重定义平台对象
- 重定义 AI 子引擎能力
- 重定义技术栈

## 读完后你应该带走什么

- 交付层现在已经围绕 `doc/作品文档` 收敛成一套完整提交包。
- 文档、页面、访问、答辩四条线已经能用同一顺序讲述。
- 后续新增材料应继续遵守页数、时长和术语三条红线。
