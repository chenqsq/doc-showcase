## 与其他文档的边界

本文只负责“比赛要求怎么落到交付包”。  
不负责重新定义平台结构，也不负责替代作品文档正文。

## 一句话先记住

> 最稳的比赛口径是：对外讲知脉课堂，对内回链 AI主导学习平台与 AI教师子引擎，不让叙事包装和研发真源互相打架。

## 1. 对外作品定义

统一作品名：

`知脉课堂：面向高校课堂的全模态知识重构与自适应伴学智能体`

统一一句话介绍：

> 知脉课堂面向高校课堂场景，通过课堂资料上传后的自动知识重构，联动学生伴学、错题复练与教师学情洞察，形成可演示、可访问、可扩学科的单机智能体系统。

## 3. 评审最关注的 5 条链路

比赛现场必须稳定展示：

1. 课堂资料上传
2. 自动知识重构
3. 学生伴学会话
4. 错题变式复练
5. 教师学情洞察

说明：

- 这 5 条链路必须分别能在页面、文档和 API 里找到对应位置
- 第一屏必须直接进入工作台，不做营销式首页

## 4. 页面要求对齐

比赛版固定 6 个核心页面：

1. 课堂工作台首页
2. 课堂资料上传与重构页
3. 学生伴学会话页
4. 错题与复习页
5. 教师洞察页
6. 管理与接入配置页

每页都要写清：

- 目标用户
- 核心任务
- 主要模块
- 关键状态
- 异常状态
- 评委演示点
- 桌面端主布局
- 移动端降级策略

## 5. 技术选型对齐

正式比赛口径如下：

- 前端：Vue 3 + TypeScript + Vite + Vue Router + Pinia + Naive UI + Tailwind CSS + VueUse Motion + ECharts
- 后端：Go 1.24 + Gin + pgx/sqlc + PostgreSQL 16
- AI：腾讯 ADP + HTTP SSE V2
- 存储：腾讯 COS，兼容 MinIO
- 高可用：Caddy/Nginx + systemd + 健康检查 + 优雅重启 + 备份恢复 + 降级输出

## 6. 推荐答辩定位

现场推荐统一说法：

- 这不是单点解题助手，而是高校课堂智能体系统
- 高等数学是第一门完整示范学科，不是唯一学科
- 前端工作台展示学生、教师、管理三类视角
- 后端采用单体架构，强调实用、稳定、易维护

## 7. 需要避免的旧口径

- 不再对外使用“开发文档入口”作为首页主称呼
- 不再把“学生单端自主学习平台”作为唯一产品定位
- 不再把高等数学写成产品唯一边界
- 不再把微服务、Redis、MQ 写成 v1 前置要求

## 读完后你应该带走什么

- 比赛要求已经被拆解进作品文档、页面、API、访问手册和技术文档里。
- 对外和对内现在是双层包装、单套真源。
- 之后新增材料都应优先挂到 `doc/作品文档`，避免再次分叉。
