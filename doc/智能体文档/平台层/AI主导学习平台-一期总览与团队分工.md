## 与其他文档的边界

本文只负责回答 4 个问题：

1. 一期到底交付什么
2. `P0 / P1 / P2` 在这一轮里各自承担什么
3. 三个开发成员分别做什么、交什么、怎么交接
4. 最终如何收口到 `master` 与 GitHub Pages

本文不替代对象字段真源、不展开 ADP 节点配置细节，也不替代学科知识资产本身。

## 一句话先记住

> 一期交付不是“先做完 `P0` 再考虑 `P1 / P2`”，而是围绕学生主闭环、教师侧增强、工作流产品化三条工作线并行收口，最后由同一站点、同一套文档、同一次发布统一对外。

## 1. 一期交付的统一定义

一期固定打包交付下面 4 个公开入口：

1. 一期总览与团队分工
2. 高等数学知识库接入与落库规范
3. Agent 工作流联调与验收手册
4. 高等数学 Agent 提示词模板与分层教学规范

这 4 个入口同时承担两件事：

- 对内作为团队执行入口
- 对外作为评委、老师和协作者的公开说明入口

## 2. 一期内并行推进的三条工作线

### 2.1 `P0` 学生主闭环稳定线

`P0` 是一期必须跑通的学生底座，负责把：

- 学习会话
- 当前任务卡
- 诊断
- 讲解
- 练习与达标判断
- 复盘与下一步

串成稳定闭环。

### 2.2 `P1` 教师侧与展示增强线

`P1` 在不破坏 `P0` 的前提下，补齐：

- 教师侧风险观察
- 班级/章节薄弱点摘要
- 补讲建议
- 学生结果可视化与展示口径

### 2.3 `P2` 工作流接入与产品化联调线

`P2` 负责把工作流与平台接口做成可持续交付的能力，包括：

- ADP 多 Agent 工作流
- 变量透传
- 检索标签绑定
- 回归样例
- `master` 收口与 GitHub Pages 对外发布

## 3. 为什么要用“一期并行”口径

因为这一轮比赛和演示的真实目标不是做出三个相互等待的阶段，而是：

- 让学生侧闭环能演示
- 让教师侧增强能解释
- 让知识库和提示词能支撑 Agent 稳定回答
- 让最终成果能公开访问

所以 `P0 / P1 / P2` 仍然保留，但只代表能力分层，不再代表必须串行排队。

## 4. 三人固定分工

### 4.1 你：项目负责人、前后端与教师侧负责人

你的责任边界固定为：

- 前端与后端落地
- 项目进度推进
- 技术方案拍板
- 教师侧产品口径
- 演示与答辩
- 提示词策略最终判断

#### 你的输入

- 知识库负责人提供的素材台账、OCR 校对稿与结构化资产
- 工作流负责人提供的 Agent 路由、变量设计与联调结果
- 当前站点、仓库和发布链路

#### 你的输出

- 首页与阅读站点的最终入口形态
- 教师侧展示口径与演示脚本
- 技术方案统一口径
- 最终发布到 `master` 和 GitHub Pages 的对外交付版本

#### 你的交接点

- 从开发 2 接知识资产成品和标签规范
- 从开发 3 接工作流配置结果和回归样例
- 统一检查首页四卡、阅读链路、公开链接和答辩话术

#### 你的验收标准

- 首页四张核心卡可直达对应文档
- 教师侧说明与演示口径统一
- `master` 分支内容可成功部署到 GitHub Pages

### 4.2 开发 2：OCR 与知识库资料电子化负责人

开发 2 的任务不是“随便把资料放进去”，而是把纸质教材、试卷、讲义变成可检索、可复用、可追踪的知识资产。

#### 开发 2 的责任范围

- 原始素材登记
- 扫描或拍照
- OCR 初稿生成
- 公式、题号、图表人工校对
- 按知识卡粒度拆分
- 元数据与标签补齐
- 批次入库
- 抽样问答验收

#### 开发 2 的输入

- 教材、试卷、讲义、PPT、板书、老师讲解资料
- 项目统一的模块编号、章节编号、资源类型规范
- 项目负责人给出的优先入库波次

#### 开发 2 的输出

- 素材台账
- OCR 校对稿
- 结构化知识资产
- 标签规范表
- 入库批次记录
- 抽样验收记录

#### 开发 2 的交接点

- 向你交付已可检索的正式资产，而不是原始扫描件
- 向开发 3 交付结构化标签和可检索字段，便于绑定检索器

#### 开发 2 的验收标准

- 资料不再只停留在纸质或图片形态
- 每批入库有统一命名、字段、状态标记
- 至少能通过抽样问答命中目标模块与章节

### 4.3 开发 3：工作流与联调负责人

开发 3 负责把文档和提示词变成真正能工作的 Agent 运行链，而不是停留在纸面设计。

#### 开发 3 的责任范围

- ADP 多 Agent 工作流配置
- `TeacherOrchestrator` 路由编排
- 各 Agent 输入输出变量定义
- 检索标签绑定
- 回归样例设计
- 联调记录
- 发布链路联通

#### 开发 3 的输入

- 项目负责人确认的提示词方向和演示重点
- 开发 2 交付的结构化知识资产与元数据
- 现有平台接口、站点和部署方式

#### 开发 3 的输出

- 可运行的 Agent 工作流
- 联调与回归记录
- 变量透传约定
- 检索标签绑定规则
- 部署与发布联通说明

#### 开发 3 的交接点

- 向你交付已联调通过的工作流与回归结论
- 向开发 2 反馈哪些标签与字段会影响检索效果

#### 开发 3 的验收标准

- 多 Agent 能按统一状态槽位工作
- 关键场景能稳定通过回归样例
- 最终发布链路不依赖临时手工解释

## 5. 三条工作线与三人如何对应

| 工作线 | 主负责人 | 协同人 | 核心产出 | 通过标准 |
| --- | --- | --- | --- | --- |
| `P0` 学生主闭环稳定线 | 你 | 开发 2、开发 3 | 学生任务推进、讲解、练习、复盘主链 | 学生闭环可完整演示 |
| `P1` 教师侧与展示增强线 | 你 | 开发 3 | 教师摘要、风险提示、展示入口 | 教师侧口径清晰且不冒充真实统计 |
| `P2` 工作流接入与产品化联调线 | 开发 3 | 你、开发 2 | Agent 工作流、变量透传、检索绑定、发布联调 | 工作流可跑、发布可验 |
| 知识库电子化与落库支撑 | 开发 2 | 你、开发 3 | OCR、拆卡、标签、入库批次 | 检索可命中、知识资产可复用 |

## 6. 一期的协作节奏

### 6.1 每一轮最小协作闭环

1. 你先给出本轮要打通的模块、教师侧目标和提示词重点。
2. 开发 2 按模块准备资料台账、OCR 校对稿和结构化资产。
3. 开发 3 用最新资产绑定检索、配置工作流并跑回归样例。
4. 你在站点上统一展示、验收并收口为对外交付口径。

### 6.2 默认交接顺序

| 顺序 | 交接内容 | 发送方 | 接收方 |
| --- | --- | --- | --- |
| 1 | 本轮模块与目标 | 你 | 开发 2、开发 3 |
| 2 | 素材台账与结构化资产 | 开发 2 | 你、开发 3 |
| 3 | 工作流配置与回归结果 | 开发 3 | 你 |
| 4 | 首页、文档、演示口径与发布 | 你 | 对外版本 |

## 7. 一期的最终发布动作

一期固定按下面顺序收口：

1. 在开发分支完成文档、提示词、首页和联调改动
2. 使用中文提交信息按主题整理提交
3. 合并到 `master`
4. 在 `master` 上执行 `npm run deploy:pages`
5. 校验 GitHub Pages 可访问且首页已显示四张核心卡

## 读完后你应该带走什么

- `P0 / P1 / P2` 在这一轮里是并行工作线，不是串行排队阶段。
- 三个开发成员不是平均分摊，而是围绕知识资产、工作流、前后端与展示形成明确交接。
- 一期最终必须统一收口到公开站点和 GitHub Pages，而不是停在本地文档或 ADP 截图。
