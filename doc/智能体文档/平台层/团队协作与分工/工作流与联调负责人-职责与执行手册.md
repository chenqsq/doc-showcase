# 工作流与联调负责人-职责与执行手册

> 文档层级：平台层  
> 文档目的：明确工作流负责人在一期中的职责、执行顺序、输入输出和通过标准  
> 核心结论：工作流负责人不是只把 Agent 接起来，而是要保证状态槽位、检索边界、变量透传和回归样例都可复用、可追踪、可验收  
> 目标读者：工作流开发者、项目负责人、协作成员

> 下游引用：ADP 配置、联调记录、回归样例、发布前检查清单  
> 适用范围：多 Agent 工作流配置、变量透传、检索绑定、回归测试与发布前联调

## 与其他文档的边界

本文只负责回答：

- 工作流负责人具体做什么
- 联调先接什么，后接什么
- 知识资产和提示词规范如何接到 Agent 流程里
- 最后怎样证明工作流真的通过了

本文不替代 OCR 手册，也不替代站点前端实现说明。

## 1. 角色定位

工作流与联调负责人在这一轮里承担 4 件事：

1. 把 Agent 职责边界定清。
2. 把变量透传和状态槽位定清。
3. 把知识库检索条件与标签绑定好。
4. 用回归样例证明这条链路可复现、可验收。

## 2. 你负责什么

- 配置或维护 ADP 多 Agent 工作流。
- 明确不同请求由哪个 Agent 处理。
- 将知识库负责人交来的标签和字段变成检索条件。
- 产出联调记录、回归样例和通过标准。
- 在发布前协助项目负责人完成最终联调。

## 3. 每轮怎么做

### 3.1 先接统一口径

在动工作流之前，先确认：

- 学生状态槽位有哪些
- 四类回答模板怎么区分
- 哪些知识卡优先给学生问答
- 哪些知识卡优先给教师摘要

### 3.2 再接知识资产

拿到 OCR 负责人交来的正式批次后，先做：

1. 检查字段是否完整。
2. 检查标签是否能区分学生问答、教师运营、课堂重构。
3. 确认每类场景的检索范围。

### 3.3 然后接 Agent 分工

把下面 6 个核心 Agent 的边界接清：

- `TeacherOrchestrator`
- `DiagnosisAgent`
- `ExplanationAgent`
- `PracticeEvalAgent`
- `ReviewPlanAgent`
- `TeacherOpsAgent`

联调时重点检查：

- 谁负责识别状态
- 谁负责解释
- 谁负责练习和达标
- 谁负责复盘
- 谁负责教师摘要

### 3.4 最后做回归

至少准备下面几类回归样例：

1. `0基础补桥`
2. `常规提分`
3. `刷题纠错`
4. `教师运营摘要`
5. 检索不到资料时的降级回复

## 4. 你的输入是什么

你需要拿到：

- 项目负责人给出的演示场景和前端展示要求
- 知识库负责人交来的结构化资产和标签规则
- 统一状态槽位和回答模板规范
- 当前平台字段和上下文透传方式

## 5. 你的输出是什么

你需要交出：

1. 可复用的 Agent 路由说明
2. 变量透传表
3. 检索绑定说明
4. 回归样例和失败样例
5. 发布前联调通过记录

## 6. 你交接给谁

### 6.1 交给项目负责人

- 哪些场景已经通过
- 哪些场景还缺知识资产或提示词约束
- 前端和答辩里要展示哪一段流程

### 6.2 反向交给 OCR 负责人

- 哪些字段和标签实际检索不到
- 哪些资料颗粒度过粗，导致回答不稳定
- 哪些章节还缺支撑样例

## 7. 验收标准

工作流负责人的工作完成，至少满足：

1. 每个 Agent 的职责边界能一句话说清。
2. 学生状态槽位能从上游一路传到最终回答。
3. 检索条件与知识资产标签一一对应。
4. 四类核心场景有可复现的回归样例。
5. 失败样例能解释原因，不是假装通过。
6. 发布前联调结果已经同步给项目负责人。

## 8. 常见失误

- 没有统一状态槽位，导致每个 Agent 说法都不一样。
- 知识库标签没有和检索条件对齐，导致命中漂移。
- 只测成功样例，不测边界和失败样例。
- 联调结果只留在口头，没有形成文档和记录。
