# 项目负责人-职责与执行手册

> 文档层级：平台层  
> 文档目的：明确项目负责人在一期中的职责边界、执行顺序、交接动作和验收标准  
> 核心结论：项目负责人不是单纯“催进度的人”，而是负责把前后端、文档口径、教师侧展示、发布链路和最终对外交付统一起来的人  
> 目标读者：项目负责人、协作成员、答辩准备者

> 下游引用：首页实现、平台文档、教师侧展示、GitHub 发布记录  
> 适用范围：一期内的前后端开发、站点结构、技术方案、进度管理与最终发布

## 与其他文档的边界

本文只负责回答：

- 项目负责人到底要管哪些事情
- 每轮开发先做什么、后做什么
- 需要别人给你什么输入
- 你最终必须交出哪些结果

本文不替代 OCR 细则和工作流联调手册。

## 1. 角色定位

项目负责人在这一轮里同时承担 5 个角色：

1. 前后端实现者
2. 技术方案拍板者
3. 进度与交接组织者
4. 教师侧与演示口径负责人
5. 最终发布负责人

## 2. 你具体负责什么

### 2.1 产品与界面

- 决定首页入口结构。
- 保证知识库页、平台文档页、阅读页和搜索体验能串起来。
- 把阅读主题、响应式、导航和入口文案统一到同一套视觉语言里。

### 2.2 技术方案

- 给知识库负责人明确命名、字段、批次和验收规则。
- 给工作流负责人明确状态槽位、数据边界和前端需要消费的结果。
- 保证文档、知识资产和工作流描述口径一致。

### 2.3 教师侧与答辩

- 统一教师侧展示内容。
- 统一“学生怎么学、教师怎么看、平台怎么联调”的讲述顺序。
- 把站点内容组织成评委一眼能看懂的入口。

### 2.4 发布

- 审查 `master` 分支的最终内容。
- 用中文提交信息整理本轮改动。
- 负责执行 `npm run deploy:pages`，并确认公网可见。

## 3. 每轮开发怎么做

### 3.1 周初或轮次开始

1. 确认本轮交付的范围。
2. 给出三个岗位各自的目标和边界。
3. 确定首页和文档入口怎么呈现。

### 3.2 周中或轮次中段

1. 跟进 OCR 负责人当前批次做到哪一步。
2. 跟进工作流负责人当前联调卡在什么地方。
3. 及时调整前端、文档和演示入口，避免最后才拼接。

### 3.3 周末或轮次收口

1. 拉通首页、文档、知识库示例和工作流说明。
2. 本地构建并检查公网展示路径。
3. 整理中文提交，推送 GitHub。
4. 合并 `master` 并完成 Pages 发布。

## 4. 你的输入是什么

你需要拿到下面几类输入：

- OCR 负责人提供的素材台账、OCR 校对稿、结构化资产和批次说明。
- 工作流负责人提供的 Agent 分工、联调结果、回归样例和失败样例。
- 站点当前代码、路由结构、阅读页组件和发布脚本。
- 本轮演示目标、答辩重点和教师侧展示要求。

## 5. 你的输出是什么

你最终必须交出：

1. 可访问的首页、知识库示例页、平台文档页和阅读页。
2. 清晰的团队分工与知识库主文档入口。
3. 教师侧展示与答辩使用的统一口径。
4. 经过验证的 `master` 分支与 GitHub Pages 链接。

## 6. 你交接给谁

### 6.1 交给 OCR 负责人

- 最新的命名和字段规范
- 当前优先要处理的资料范围
- 抽样问答与验收方式

### 6.2 交给工作流负责人

- 当前要联调的页面和场景
- 站点前端需要展示的结果格式
- 最终演示时必须跑通的路径

### 6.3 交给最终评审和协作者

- 一个能打开就看懂结构的首页
- 一组能直达核心文档的入口
- 一个公网可访问的发布版本

## 7. 验收标准

项目负责人的工作完成，至少要满足：

1. 首页结构与文档入口不再混乱。
2. 三个岗位的文档和实际协作动作一致。
3. 站点在不同主题和不同宽度下都能正常阅读。
4. 搜索能找到分工文档和知识库主文档。
5. `npm run build` 通过。
6. GitHub 仓库提交信息为中文。
7. 公网地址打开后能看到最新首页。

## 8. 常见失误

- 只顾前端界面，没有把知识库和工作流的输入输出提前对齐。
- 到最后一天才收文档，导致首页入口和实际内容对不上。
- 只发布代码，不核对公网展示内容是否已经更新。
- 让答辩口径和站点实际入口不一致。
