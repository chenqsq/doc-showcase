# 团队协作与执行分工

> 文档层级：平台层  
> 文档目的：把团队分工、交接和用 AI 推进的方法压缩成一篇比赛可执行说明  
> 核心结论：这轮不是做复杂组织设计，而是让项目负责人、知识库负责人、工作流与联调负责人三个人同时知道自己要交什么  
> 目标读者：团队成员、答辩准备者  
> 推荐下一步：需要比赛视角表达时，继续看 [比赛交付与答辩手册.md](../交付层/比赛交付与答辩手册.md)

## 1. 这轮只分 3 个执行角色

| 角色 | 只抓什么重点 | 最后交什么 |
| --- | --- | --- |
| 项目负责人 | 收口站点、文档、演示、发布 | 一个能看懂、能访问、能演示的版本 |
| 知识库负责人 | 整理资料、拆知识资产、补标签 | 一批可检索、可交接的知识内容 |
| 工作流与联调负责人 | 让 AI 流程跑通、测稳、记清问题 | 一组通过样例和失败说明 |

## 2. 每个人怎么用 AI

### 2.1 项目负责人

- 用 AI 压缩展示重点和首页文案。
- 用 AI 整理演示顺序和发布检查清单。
- 人工负责确认站点内容和真实实现一致。

### 2.2 知识库负责人

- 用 AI / OCR 做初步识别和清洗。
- 用 AI 帮忙拆知识点、生成标题和关键词候选。
- 人工负责校对公式、题号、图表和来源。

### 2.3 工作流与联调负责人

- 用 AI 梳理流程、补测试样例、归纳失败原因。
- 用 AI 辅助整理提示词草稿和回归清单。
- 人工负责验证链路是否真的跑通。

## 3. 最少怎么交接

### 3.1 项目负责人 -> 知识库负责人

- 本轮优先章节
- 命名和交付标准
- 比赛必须支持的展示问题

### 3.2 知识库负责人 -> 工作流与联调负责人

- 已整理好的知识内容
- 标题、来源、关键词和标签
- 可直接测试的示例问法

### 3.3 工作流与联调负责人 -> 项目负责人

- 已跑通场景
- 未跑通场景
- 失败原因
- 前端应该展示的结果

## 4. 收口时看什么

1. 首页和推荐阅读已经统一。
2. 平台、子引擎和高数示例说的是同一条主线。
3. 比赛演示顺序和站点入口一致。
4. `npm run build` 通过。
5. GitHub Pages 上已经是最新内容。

## 5. 这篇为什么比旧岗位手册更短

因为这次我们只保留真正会影响协作的东西：

- 谁负责什么
- 怎么用 AI 提速
- 最少交接什么
- 最后怎么收口

那些更细的实现内容已经被移到平台规范、工作流手册和高数实施文档里，不再重复写三遍。
