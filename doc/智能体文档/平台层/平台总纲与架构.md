# 平台总纲与架构

> 文档层级：平台层  
> 文档目的：用一篇文档收住项目定位、平台主线、角色分工和整体架构  
> 核心结论：平台真正成立，不是因为某个 Agent 会回答，而是因为平台、子引擎、学科知识库和比赛交付能沿同一条主线协作  
> 目标读者：新成员、实现者、评委、答辩准备者  
> 推荐下一步：实现细节继续看 [平台对象、生命周期与验收.md](./平台对象、生命周期与验收.md)

## 1. 这到底是个什么项目

这是一个 AI 主导学习平台的公开演示站点。

它不只展示“一个 AI 老师会回答问题”，而是展示四件事一起成立：

1. 学生能沿一条连续学习链路被推进。
2. 教师能看到结构化结果并介入。
3. 知识库不是杂乱资料，而是可检索、可扩科的资产体系。
4. 平台、子引擎和学科示例能在同一站点中被讲清楚。

## 2. 平台为什么要按 5 条主阅读线组织

过去的问题不是没有内容，而是所有内容都按作者分层堆在一起：

- 平台层讲平台
- 子引擎层讲 Agent
- 学科层讲高数
- 交付层讲比赛

这样写对作者方便，对读者不方便。

现在改成 5 条主阅读线：

1. 先读项目
2. 平台设计
3. 子引擎实现
4. 高数示例
5. 比赛交付

好处是第一次进入时不用先理解目录结构，只要知道自己接下来要解决什么问题。

## 3. 当前平台的主线是什么

平台主线可以压成一句话：

> 用统一对象组织学习过程，用子引擎执行教学，用知识库承接内容，用学科示例验证平台成立。

对应到站内，分别是：

- 平台对象和生命周期负责定义“平台怎么看学习”。
- 子引擎负责定义“AI 怎么执行这轮教学”。
- 知识库与学科接入负责定义“内容怎么被组织和命中”。
- 高数示例负责证明“这套机制在真实学科里能跑”。

## 4. 现在有哪些固定角色

### 4.1 平台视角

- 平台层负责项目主线、对象、验收和知识库接入规则。
- 子引擎层负责 AI 教学执行、状态识别、检索调用和联调回归。
- 学科层负责把平台规则落到一门具体学科上。
- 交付层负责把整套内容翻译成比赛叙事与答辩口径。

### 4.2 团队执行视角

- 项目负责人负责收口站点、文档、演示和发布。
- 知识库负责人负责整理知识内容。
- 工作流与联调负责人负责让 AI 链路跑通。

团队执行细节看 [团队协作与执行分工.md](./团队协作与执行分工.md)。

## 5. 架构怎么理解最省事

### 5.1 第一层：平台

平台不直接替你“讲题”，它主要负责三件事：

- 定义对象
- 组织阅读和展示
- 规定接入与验收口径

### 5.2 第二层：子引擎

子引擎负责把一轮教学真正执行出来，包括：

- 状态判断
- 讲解
- 练习
- 复盘
- 教师侧摘要

### 5.3 第三层：学科与知识资产

高等数学作为第一门示范学科，负责证明：

- 一门完整课程可以被组织成知识资产
- AI 可以在这门课里稳定工作
- 这套结构以后可以迁移到别的学科

## 6. 为什么还保留 P0 / P1 / P2

`P0 / P1 / P2` 现在不再代表必须串行完成的阶段，而是并行推进的三条工作线：

- `P0` 学生主闭环
- `P1` 教师侧和展示增强
- `P2` 产品接入和联调发布

它们的实施细节收在 [AI教师子引擎实施附录-P0-P2.md](../子引擎层/AI教师子引擎实施附录-P0-P2.md)。

## 7. 当前成功标准是什么

1. 首页能把整套项目结构讲清楚。
2. 平台对象、子引擎和学科示例之间没有各讲各的。
3. 高数示例能证明这不是空叙事，而是有真实资产和可跑链路。
4. 比赛交付能把复杂实现翻译成评委听得懂的话。

## 8. 读完后你应该带走什么

- 这是一个平台型项目，不是单个答题 Agent 演示。
- 平台、子引擎、学科示例和比赛交付各有位置，但要说同一套话。
