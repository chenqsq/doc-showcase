# AI主导学习平台-平台需求与验收

> 文档层级：平台层  
> 文档目的：把平台能力抽象成需求与验收口径，并加入角色向通过标准  
> 核心结论：平台验收看的不是“某轮回答好不好”，而是“角色主线是否成立、对象链路是否闭合、阶段能力是否可继承、竞争力是否被保留”  
> 目标读者：产品负责人、方案设计者、研发协作者、答辩准备者  
> 上游真源：[AI主导学习平台-角色主线与阶段地图.md](./AI主导学习平台-角色主线与阶段地图.md)、[AI主导学习平台-统一对象与接口契约.md](./AI主导学习平台-统一对象与接口契约.md)  
> 下游文档：[AI主导学习平台-学习生命周期与编排策略.md](./AI主导学习平台-学习生命周期与编排策略.md)、[AI主导学习平台-总体架构设计.md](./AI主导学习平台-总体架构设计.md)、[AI教师子引擎-PRD.md](../子引擎层/AI教师子引擎-PRD.md)  
> 适用范围：平台需求、阶段通过标准、角色向验收口径

## 与其他文档的边界

本文定义平台需求与验收。  
角色分工以角色主线文档为准，对象字段以对象契约文档为准；本文只把它们翻译成必须通过的需求与验收口径。

## 一句话先记住

> 平台验收不是只看“学生问一句，系统答一句”，而是看学生能不能被持续推进、教师能不能看到风险、学科能不能按模板接入、接入方能不能稳定透传上下文。

## 1. 一页结论

平台最低成立条件，不是“能答题”，而是能稳定跑通下面这条主链路：

`学习档案 -> 学习会话 -> 当前任务卡 -> 子引擎教学执行 -> 子引擎回流结果 -> 课节笔记 -> 个人总复习本 -> 教师运营摘要 -> 学科接入 -> 产品接入与沉淀`

## 2. 平台功能需求（PFR）

| 编号 | 能力 | 目标 | 关键输出 |
| --- | --- | --- | --- |
| `PFR-01` | 学习建档 | 固定学生起点、目标与初始阶段 | 学习档案 |
| `PFR-02` | 学习会话续接 | 固定当前轮次上下文与历史连续性 | 学习会话 |
| `PFR-03` | 当前任务锁定 | 明确本轮目标、完成标准、回补条件 | 当前任务卡 |
| `PFR-04` | 学生持续推进 | 不依赖学生先会提问，也能继续往前学 | 推进或回补动作 |
| `PFR-05` | 双层笔记沉淀 | 每轮学习结束后沉淀可复习资产 | 课节笔记、个人总复习本 |
| `PFR-06` | 教师运营支持 | 聚合风险、趋势与干预建议 | 教师运营摘要 |
| `PFR-07` | 学科接入与扩科 | 新学科可按模板挂入平台 | 学科接入模板、学科示范 |
| `PFR-08` | 产品接入与外部集成 | 稳定接入前端/后端并透传上下文 | `BFF`、`HTTP SSE`、接入字段 |
| `PFR-09` | 学习记录沉淀 | 让结果进入可追踪、可运营的数据层 | 学习记录、聚合输出 |

## 3. 平台非功能需求（PNFR）

| 编号 | 类别 | 要求 |
| --- | --- | --- |
| `PNFR-01` | 边界清晰 | 平台层、子引擎层、学科层、交付层职责清晰 |
| `PNFR-02` | 角色可解释 | 读者能清楚说出每个正式角色的主线与阶段分工 |
| `PNFR-03` | 对象可追溯 | 任一核心对象都能回链到统一对象契约文档 |
| `PNFR-04` | 阶段可继承 | `P0 / P1 / P2` 能说明继承关系与非替代关系 |
| `PNFR-05` | 扩科可复用 | 新学科接入时不重写平台机制 |
| `PNFR-06` | 接入可连续 | 外部接入时上下文连续、身份连续、记录可沉淀 |
| `PNFR-07` | 竞争力不降级 | 多学科、教师主线、`BFF`、`HTTP SSE`、自定义前端、学习记录沉淀都仍在现行主文档中 |

## 4. 平台验收项（PAC）

| 编号 | 对应需求 | 通过标准 |
| --- | --- | --- |
| `PAC-01` | `PFR-01` | 学生首次进入后能形成学习档案，而不是落到空白聊天框 |
| `PAC-02` | `PFR-02` | 同一学生再次进入时能续接学习上下文，而不是孤立重开 |
| `PAC-03` | `PFR-03` | 每一轮学习都有当前任务卡，明确目标、标准和回补条件 |
| `PAC-04` | `PFR-04` | 学生不主动提问时，平台仍能至少推进一轮或明确触发回补 |
| `PAC-05` | `PFR-05` | 每轮结束后生成课节笔记，并同步更新个人总复习本 |
| `PAC-06` | `PFR-06` | 教师侧至少能看到风险信号、停滞点或干预建议 |
| `PAC-07` | `PFR-07` | 高等数学之外的新学科能按模板挂入，而不重改平台总纲 |
| `PAC-08` | `PFR-08` | 接入方能稳定托管 `AppKey`、固定 `visitor_biz_id`、透传 `custom_variables`、使用 `HTTP SSE` |
| `PAC-09` | `PFR-09` | 学习结果能沉淀为后续可追踪、可聚合、可复用的记录 |

## 5. 角色向验收

### 5.1 学生主线

- 能否知道自己当前在哪个阶段、模块、课节
- 能否知道这一轮为什么学这个
- 能否在未达标时看到明确回补路径
- 能否在每轮结束后看到沉淀结果

### 5.2 教师/运营者主线

- 能否看到风险学生与停滞点
- 能否看到趋势摘要而不是原始聊天堆叠
- 能否看到可执行干预入口

### 5.3 平台系统主线

- 能否稳定维持学习档案、学习会话、当前任务卡、双层笔记这条链
- 能否把子引擎回流结果转为推进/回补决策

### 5.4 AI教师子引擎主线

- 能否稳定承接学生教学执行线
- 能否输出教师运营支持线所需摘要
- `TeacherOpsAgent` 是否保持增强旁路，不阻塞学生主闭环

### 5.5 产品接入方主线

- 能否稳定透传 `visitor_biz_id`、`custom_variables`、`chapter_id`、`role`
- 能否以 `BFF` 和 `HTTP SSE` 形式接入，而不重造平台
- 能否沉淀学习记录并为教师侧聚合留接口

## 6. 阶段向验收

| 阶段 | 本阶段必须成立 | 本阶段不要求替代 |
| --- | --- | --- |
| `P0` | 学生主闭环底座、对象主链、基础上下文连续性 | 不替代教师运营增强和产品化接入 |
| `P1` | 学生可视化、教师运营增强、`TeacherOpsAgent` 旁路 | 不替代 `P0` 主底座，不要求产品后端成为前置依赖 |
| `P2` | 产品后端/BFF、`HTTP SSE`、自定义前端、学习记录沉淀 | 不替代 ADP 内部编排，不要求重后端或微服务化 |

## 7. 一致性验收

平台现行文档必须同时满足：

- 任一核心术语只保留一个正式定义源
- 活跃文档引用对象时能回到统一对象契约
- 活跃文档引用角色时能回到角色主线与阶段地图
- 交付层明确声明自己只做下游翻译

## 读完后你应该带走什么

- 平台验收是链路验收，不是单点功能验收。
- 角色向验收和阶段向验收都已经进入正式口径。
- `P1 / P2` 的竞争力必须继续在现行主文档里存在。
